# FRACTAL CORE: ONTOLOGY & CQRS ARCHITECTURE MANIFEST
## 1. THE RELATIONAL-JSON HYBRID MODEL (CQRS)
This system operates on a strict Separation of Concerns between Persistence and Presentational logic.
*   **SQL (Persistence):** SQLite is strictly used as a high-speed, dumb persistence layer. It ensures Referential Integrity via Foreign Keys but MUST NEVER execute Business Logic, Calculations, or Complex Aggregations.
*   **JSON (Presentation/Flexibility):** Entity attributes (Property Sets / PSets) are stored as raw JSON blobs in `dat_pset_live_payloads`. 
*   **RULE 1.A:** DO NOT CREATE TRIGGERS OR STORED PROCEDURES FOR CALCULATIONS (e.g., summing compositional arrays). All mathematical logic, array iteration, and JSON validations MUST run in the NodeJS backend or React frontend. SQLite only stores the final string.
*   **RULE 1.B (OPTIMISTIC CONCURRENCY):** To prevent JSON Lost Updates in multi-user environments, all JSON payloads MUST be strictly versioned (`__v` or `ETag`) by the backend. The backend MUST reject any update where the client's `__v` does not match the Database's current `__v` using an HTTP 409 Conflict.

## 2. THE L-MATRIX ENTITY ONTOLOGY (Revit/BIM Parametric Equivalency)
Entities exist simultaneously across a deterministic, 4-tier acyclic mathematical hierarchical graph to prevent contextual AI hallucination:
1.  **L1 CATEGORY (`def_entity_l1_category`):** The Root Concept (e.g., EMPRESA). Hardcoded AI behavior anchor.
2.  **L2 FAMILY (`def_entity_l2_family`):** The Business Division (e.g., EMP-UTE or EMP-NOUTE).
3.  **L3 TYPE (`def_entity_l3_type`):** The concrete Mold (e.g., UTE-PAJARES-TYPE).
4.  **L4 INSTANCE (`dat_entity_l4_instance`):** The live, physical record representing the real-world object (e.g., "Nortunel S.A."). Instances clone the L3 blueprints requirements but hold their own overriding data values in `dat_pset_live_payloads`.

## 3. PROPERTY SET (PSet) MECHANICS & RBAC SECURITY
Do not normalize properties into individual SQL columns. Properties function as isolated `JSON Schemas`.
Each PSet dictionary (`def_pset_template.json_shape_definition`) MUST separate Data Topology from UI Behavior.
It must contain two distinct internal branches:
*   `DataSchema`: Defines the physics (type, range, enumerators, default values).
*   `UISchema`: Defines the interaction security:
    *   `ui:readonly`: (Boolean) If true, acts as a "Type Parameter". The Instance inherits the value but cannot mutate it. If false, it acts as an "Instance Parameter".
    *   `security:visible_roles`: Array defining RBAC filtering. `[CREADOR, ADMINISTRADOR, USUARIO]`.

**RULE 3.A (SECURITY PRUNING / ZERO BROWSER TRUST):**
The Frontend (React) MUST NEVER be trusted to enforce `visible_roles` visibility by merely hiding visual React components. 
The NodeJS Backend MUST actively parse the `UISchema`, evaluate the requesting user's role against `visible_roles`, and physically **DELETE (PRUNE) the restricted JSON keys** from the payload BEFORE transmitting the HTTP response over the network. Restricted data (salaries, admin flags) must NEVER reach the browser's Network Tab in plain text.

**THE THREE OPERATIVE HIERARCHIES (ROLES):**
1.  **CREADOR (System Level):** Owns architecture. Assigns immutable base PSets (System Parameters). Example: `soyUte: true` (Read-only).
2.  **ADMINISTRADOR (Project Level):** Owns `def_pset_template` (User PSets). Can invent new schemas dynamically and assign them to any tier (L1, L2, L3, or L4).
3.  **USUARIO (Instance Level):** Only owns `dat_pset_live_payloads` (Value injection). Can only mutate `ui:readonly: false` paths.

## 4. HANDLING GRAPH TOPOLOGY vs COMPOSITION
If asked to link Entities together, YOU MUST EVALUATE THE ARCHITECTURAL INTENT:
*   **SCENARIO A (Multidimensional / Operational Trees):** If Entity A "belongs to" or "is a child of" Entity B, and can belong to multiple entities simultaneously without destroying its identity...
    *   **SOLUTION:** Use the `rel_entity_topology_graph` M:N table. E.g., Nortunel S.A. belongs to an Operational Zone.
*   **SCENARIO B (Strict Object Composition / Arrays):** If an Entity is fundamentally composed of smaller entities inside it (e.g., a "UTE" is composed of M "NOUTE" partners + their ownership percentages)...
    *   **SOLUTION:** DO NOT USE THE TOPOLOGY GRAPH. Use an array inside a Dynamic PSet bound to the L2/L3 Blueprint via `rel_pset_to_entity_bridge`. Array shape: `[{ id_empresa, porcentaje }]`. The Backend calculates math constraints dynamically based on this array before writing to SQLite.
