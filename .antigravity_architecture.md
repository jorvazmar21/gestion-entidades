# FRACTAL CORE: ONTOLOGY & CQRS ARCHITECTURE MANIFEST
## 1. THE RELATIONAL-JSON HYBRID MODEL (CQRS)
This system operates on a strict Separation of Concerns between Persistence and Presentational logic.
*   **SQL (Persistence):** SQLite is strictly used as a high-speed, dumb persistence layer. It ensures Referential Integrity via Foreign Keys but MUST NEVER execute Business Logic, Calculations, or Complex Aggregations.
*   **JSON (Presentation/Flexibility):** Entity attributes (Property Sets / PSets) are stored as raw JSON blobs in `dat_pset_live_payloads`. 
*   **RULE 1.A:** DO NOT CREATE TRIGGERS OR STORED PROCEDURES FOR CALCULATIONS (e.g., summing compositional arrays). All mathematical logic, array iteration, and JSON validations MUST run in the NodeJS backend or React frontend. SQLite only stores the final string.
*   **RULE 1.B (OPTIMISTIC CONCURRENCY):** To prevent JSON Lost Updates in multi-user environments, all JSON payloads MUST be strictly versioned (`__v` or `ETag`) by the backend. The backend MUST reject any update where the client's `__v` does not match the Database's current `__v` using an HTTP 409 Conflict.

## 2. THE L-MATRIX ENTITY ONTOLOGY (Revit/BIM Parametric Equivalency)
Entities exist simultaneously across a deterministic, 4-tier acyclic mathematical hierarchical graph to prevent contextual AI hallucination:
1.  **L1 CATEGORY (`def_entity_l1_category`):** The Root Concept (e.g., EMPRESA). Hardcoded AI behavior anchor.
2.  **L2 FAMILY (`def_entity_l2_family`):** The Business Division (e.g., EMP-UTE or EMP-NOUTE).
3.  **L3 TYPE (`def_entity_l3_type`):** The concrete Mold (e.g., UTE-PAJARES-TYPE).
4.  **L4 INSTANCE (`dat_entity_l4_instance`):** The live, physical record representing the real-world object (e.g., "Nortunel S.A."). Instances clone the L3 blueprints requirements but hold their own overriding data values in `dat_pset_live_payloads`.

## 3. PROPERTY SET (PSet) MECHANICS & RBAC SECURITY
Do not normalize properties into individual SQL columns. Properties function as isolated `JSON Schemas`.
Each PSet dictionary (`def_pset_template.json_shape_definition`) MUST separate Data Topology from UI Behavior.
It must contain two distinct internal branches:
*   `DataSchema`: Defines the physics (type, range, enumerators, default values).
*   `UISchema`: Defines the interaction security:
    *   `ui:readonly`: (Boolean) If true, acts as a "Type Parameter". The Instance inherits the value but cannot mutate it. If false, it acts as an "Instance Parameter".
    *   `security:visible_roles`: Array defining RBAC filtering. `[CREADOR, ADMINISTRADOR, USUARIO]`.

**RULE 3.A (SECURITY PRUNING / ZERO BROWSER TRUST):**
The Frontend (React) MUST NEVER be trusted to enforce `visible_roles` visibility by merely hiding visual React components. 
The NodeJS Backend MUST actively parse the `UISchema`, evaluate the requesting user's role against `visible_roles`, and physically **DELETE (PRUNE) the restricted JSON keys** from the payload BEFORE transmitting the HTTP response over the network. Restricted data (salaries, admin flags) must NEVER reach the browser's Network Tab in plain text.

**THE THREE OPERATIVE HIERARCHIES (ROLES):**
3.  **USUARIO (Instance Level):** Only owns `dat_pset_live_payloads` (Value injection). Can only mutate `ui:readonly: false` paths.

## 4. UI ABSTRACTION PRINCIPLE (THE ILLUSIONIST RULE)
**RULE 4 (ZERO EXPOSURE OF DB FRAGMENTATION):**
The database is highly normalized and fragmented (L-Matrix, generic bridges, CQRS PSets, etc.). **THE REACT FRONTEND MUST NEVER EXPOSE THIS FRAGMENTATION TO THE USER.** 
Do NOT create tabs or visual tables mapped 1:1 to SQLite tables. 
Instead, the UI MUST dynamically group, merge, and consolidate property inputs visually, driven strictly by the `fk_ui_group` integer metadata present in the `def_pset_template` dictionary, which resolves via Foreign Key to the sorting code (`ui_group_code`) and the visual label (`ui_group_name`) living in the centralized `def_ui_groups` table. The Administrator must experience a seamless, monolithic form-like interface, while the underlying system dissects the payloads mathematically.

## 5. HANDLING GRAPH TOPOLOGY vs COMPOSITION
If asked to link Entities together, YOU MUST EVALUATE THE ARCHITECTURAL INTENT:
1. Is it a strict parent/child life-cycle dependency (like UTE and its specific composition)? Use a dedicated PSet dictionary array (`PSET_EMP_UTE_COMPO`).
2. Is it a multi-graph operational relationship (Worker assigned to 4 Machines across 2 worksites)? Use the `rel_entity_topology_graph` table connecting multiple L4 instances dynamically.

## 6. SURROGATE VINCULATION & UBIQUITOUS AUDIT (TIER-1 ARCHITECTURE)
**RULE 5 (DECOUPLED FOREIGN KEYS & SOFT DELETION):**
All foundational architectural definitions (L1 Categories, L2 Families, L3 Types, PSets, UI Groups) are bound relationally by **Surrogate Integer Primary Keys** (e.g., `id_l1=100`, `fk_ui_group=1`). 
The string-based codes (e.g., `l1_code: "L1_EMP"`) belong to the Administrator and are completely editable. NEVER rely on text strings for JOIN conditions.
Additionally, ALL foundational tables enforce **Ubiquitous Auditing**. NEVER perform a `DELETE FROM def_entity...` or `DELETE FROM def_pset...`. When destroying architecture, you MUST orchestrate a Soft-Delete (`UPDATE ... SET is_active=0, deleted_at=CURRENT_TIMESTAMP`).

## 7. MULTI-DIMENSIONAL ABAC MATRIX (PHASES & DEPARTMENTS)
**RULE 6 (N:M SECURITY ARBITRATION):**
Do not bind Phase or Department security rules directly within the JSON raw data or via dummy `fk_` strings if the Entity supports many-to-many lifecycles. 
Instead, rely on the `rel_abac_matrix_rules` relational bridging table to define the tactical DNA (`can_read`, `can_write`, `can_approve`) bounding a PSet (`def_pset_template`) to specific explicit Phases (`def_lifecycle_phase`) and Departments (`def_company_department`). Always enforce SQLite's strict Foreign Keys.

## 8. L-MATRIX PARAMETRIC INFERENCE ENGINE (WIZARDS & BULK INGESTION)
**RULE 7 (No Hardcoded L3 Foreign Keys):**
When writing Ingestion Wizards (Frontend) or Bulk Importers (Backend), NEVER hardcode logic like `if (isUte) fk_l3 = 1`. 
The correct procedural workflow to forge an L4 Instance is a Progressive Parametric Inference targeting the L3 Mold:
1. The user selects an L1 Category (e.g., EMPRESA).
2. The user inputs business parameters (e.g., `esUte = true`).
3. The system dynamically queries or filters ALL available L3 Types under that L1 Category.
4. The system compares the user's input against the static/default PSets bound to those L3 Types.
5. **Only ONE L3 Type should perfectly match the provided parameter footprint.** (e.g., the Molde L3 with a bound PSet `{"esUte": {"default": true}}`).
6. The system infers this unique `fk_l3` ID algorithmically and creates the L4 Instance cloning that specific Mold.
*Warning:* If two L3 Molds share the exact same parameter footprint, the inference engine will collide. Ensure all L3 Models are parametrically distinguishable or explicitly selectable.
